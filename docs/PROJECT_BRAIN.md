## ★ 四条底层逻辑（用户钦定，凌驾于一切实现细节之上）

以下四条是《符塔》的产品宪法。任何设计、任务、实现、验收发生冲突时，**以这四条为最高优先级**。每条都给出了可被 Codex 执行、可被验收量化的落地约束。

### 逻辑一：摸鱼场景 · 单局轻负担 · 核心操作不复杂

- **场景假设**：玩家在通勤、排队、工作摸鱼的碎片时间单手竖屏游玩。任何设计都要扛得住“随时被打断、随时能回来”。
- **硬约束**：
  - 单局主线普通关 75–105 秒、Boss 关 110–140 秒（见总设计 6 章），不得超时。
  - 核心循环只有三件套：**摆位（拖拽）/ 对位（选带哪 5 符）/ 画功（可选手绘）**。除此之外不得新增“必须实时操作”的硬核负担。
  - 自动战斗为默认态：手残/摸鱼玩家**全程不画符也能稳定推图**，零惩罚（画符是上限不是门槛）。
  - 单局必做的主动决策点 ≤ 4 个（布阵 1 次 + 灵机点 3 次）。不得堆砌入口和弹窗。
  - 新手前 3 分钟遵循“先赢一把再教系统”（FTUE，总设计 1·C），绝不一上来糊脸 10 个入口。
- **反面红线**：禁止引入连续 QTE、长按蓄力链、多指同时操作、强制不可暂停的高频微操。

### 逻辑二：一定要爽 · 三类玩家都有爽点

- **肝帝爽**：等级不封顶 + 无尽符塔 + 五行专精圆满，永远有可追的数值与排名（总设计 3.6 / 3.7）。
- **氪佬爽**：8 条养成线堆战力，越级碾压低战力内容的“数值爽”成立；但**红线：不卖一锤定音的数值碾压**（总设计 8.3）。付费买的是效率与外观，不是把免费玩家锁死。
- **免费/技术玩家爽（翻盘空间）**：靠**相生摆位 + 克制对位 + 画功**，同一张符输出可差 3 倍（总设计 5.1）；技巧最多补 **25–30%** 战力差（总设计 3.6 / 4.6），保证越级有活路、不被纯数值碾退游。
- **设计哲学一句话**：**战力打底（下限），技巧翻盘（上限）。** 实现与验收时，下限（数值咬合）和上限（技巧收益）都必须可被量化复现。

### 逻辑三：数值与代码隔离 · 配置不得硬编码

- **铁律**：数值配置、关卡配置、符/怪/经济/融合/视觉色板配置，**一律不得硬编码进业务逻辑**。业务代码只读配置、跑公式，不内嵌具体数字。
- 所有可调参数（战斗参数、20 关、怪物模板、符模板、升级消耗、融合配方、推荐战力曲线、减伤常数 K=600、爬塔底数 1.15 等）必须落在独立的配置层。
- 详细架构、目录、Schema、校验门禁见 `docs/DATA_ARCHITECTURE.md`。这是**强制门禁项**，违反即判 FAIL。

### 逻辑四：雾山五行水墨国风 · 特效走水墨/泼墨/符箓/五行色彩

- 整体 UI 与世界观美术走《雾山五行》方向：水墨、留白、线条骨法、国风。
- 技能特效统一为水墨/泼墨/符箓笔触 + 五行色彩语言（金白、木青、水墨蓝、火朱、土赭）。
- 视觉是品牌护城河，不是后期换皮。详细色板、特效规范、验收标准见 `docs/VISUAL_DIRECTION.md`。

## ★ 工程纪律（底层逻辑·增补章，2026-07-02，用户核准）

与上述四条产品宪法**同级、同为最高约束**。四条管"做什么样的游戏"，本章管"怎么干活不出事"。来源：T9 假完成事故、2026-06-28 信息注入测试、T2–T4 施工实践的成文化。

### 纪律一：不信自报，只信独立复核
施工者(Codex)的完成声明、转述的门禁文本、wrapper 的 exit code 一律不作为验收依据。**硬证据双证**：`git status --porcelain` 非空（产出真实落盘）+ 门禁由监督者**独立复跑**。施工日志中的 `apply_patch` 块仅作辅助信号，**不作硬门**（codex 日志格式随版本变化：2026-07-02 T4 实测新版日志无此字样而产出真实）。派工前先 `codex exec` 烟测返回 READY 才派。

### 纪律二：尺子不可侵犯，且由机器验证
尺子清单 = `tests/*.gate.test.ts`、`tests/golden.formulas.test.ts`、`scripts/**`（门禁/派工/平衡模拟全部）、`acceptance/**`（含 contracts.json）、`docs/**`、`package.json`。每次施工后由 `scripts/codex-build.sh` **自动**求施工 diff 与尺子清单的交集，非空即 FAIL——不依赖人工看 git status。尺子的修改权唯一归 Claude，且 Claude 改尺子须自证等价或说明变更理由。

### 纪律三：标准施工前钉死，证据事后归档
每批任务先落契约（contracts.json DoD）与尺子测试、再签发简报、最后派工——**顺序不可逆**。验收通过后，门禁输出（`reports/`）、commit hash、日志摘要须可追溯、可审计。

### 纪律四：outward 动作必须多源交叉印证
push 后以 `git rev-parse HEAD` 与远端（`git ls-remote origin main` 或 `gh api`）**完整 hash** 比对一致才算成功。任何关键结论不采信单一来源（含工具返回层）；发现内部矛盾绝不挑对自己有利的信；结果碰巧对 ≠ 验证到位。

### 纪律五：确定性与单一真相源
一切随机必须走注入 seed 的 RNG，禁止 `Math.random`/`Date` 参与模拟——同 seed 必须逐字节复现（回放/埋点/平衡校准的根基）。任何数据只允许一个权威来源：**数值 = balance-sim KNOBS、几何 = maps.json、视觉 = visual.json**；凡出现两处人肉同步即为债，须登记并消除（T6 的 balance-sim 数值核重构即执行本条）。宪法行文中的历史数值示例（如逻辑三提及的旧 K=600、爬塔 1.15，已演进为相对式 relK=0.68、g=1.07）一律以 KNOBS 为准，不逐处追改。

### 纪律六：技术债显式登记，禁止靠记忆
已知债写入 `docs/TECH_DEBT.md`；每批验收时过一遍债表；还债也走契约流程（钉契约→施工→验收）。

### 纪律七：失败要快、要响
配置违反 schema 即抛错，禁止静默兜底/默认值救场；门禁任何一层 FAIL 即整批 FAIL，不允许"先过后补"。

## 角色分工

| Claude Code | 项目大脑、监督者、思路转换器、验收者 | `docs/`、`scripts/`、`tests/*gate*`、`acceptance/`、验收标准、任务拆解、架构方向、缺陷清单 | 不直接替 Codex 大规模写业务实现，除非是在纠偏或修复验收标准 |

| 用户 | 产品负责人、最终裁决者 | 需求、方向、取舍 | 无 |

## 工作原则

1. Claude 先把用户想法转成可执行任务、验收标准和优先级。
2. Codex 按任务实现，不自行扩大范围。
3. 每批实现完成后必须运行 `npm run quality:gate`。
4. Claude 的验收结论必须引用门禁输出、截图/录像证据、文件变更和未解决风险。
5. “Claude 说通过”不等于通过；通过必须满足 `docs/ACCEPTANCE_GATE.md` 的核准值。

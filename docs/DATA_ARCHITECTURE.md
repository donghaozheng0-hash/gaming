# 符塔 · 数值/配置与代码隔离架构

> 对应底层逻辑三：**数值与代码隔离，配置不得硬编码。** 本文是强制门禁规范，违反即判 FAIL。
> 维护：Claude（架构与 Schema）。实现：Codex（loader、校验、配置内容）。

## 1. 一句话原则

业务代码（战斗、渲染、养成、UI）**只读配置、只跑公式、只消费数据**；任何具体数字、关卡内容、符/怪/经济参数都不得写死在 `.ts` 业务逻辑里。改数值 = 改配置，不改代码、不重新编译逻辑。

## 2. 隔离三层

| 层 | 职责 | 允许出现的内容 | 禁止出现的内容 |
|---|---|---|---|
| **配置层** `src/config/**` | 纯数据 | 数值、关卡、符/怪/经济/融合表、视觉色板键 | 任何逻辑、Babylon API、分支判断 |
| **领域层** `src/game/**` | 公式与规则 | 战力/伤害/承伤公式、波次调度、克制相生判定 | 写死的具体数值常量（必须来自配置层） |
| **表现层** `src/render/**`、UI | 渲染与交互 | Babylon mesh/特效/输入 | 数值规则、平衡数字 |

> 公式里的“结构性常数”（如战力公式 `ATK×6 + HP×0.6 + DEF×3`、减伤常数 K=600、爬塔底数 1.15）也必须作为**具名配置项**放入 `balance` 配置，而不是散落在代码里的魔法数字。

## 3. 目标目录结构（Codex 落地，可微调命名但不可合并职责）

```text
src/
  config/
    schema/           # 每类配置的 TypeScript 类型 + 运行时校验
    balance.json      # 战力/伤害/承伤公式常数、相生×1.3/克制×1.5/品质倍率、减伤K
    runes.json        # 10 张符模板（攻击、攻速、射程、特性、解锁、五行）
    monsters.json     # 8 类怪模板（HP/护盾/攻击/速度，以 R 为系数）
    levels.json       # 20 关：推荐战力、波次、敌人组合、首通奖励、地图模板引用
    waves.json        # 普通关/Boss关 7 波模板
    maps.json         # 路径模板池 + 候选格位（8–10）+ 开放规则（随机 6）
    fusion.json       # 融合配方 + 合成元素克制 + 开放节奏
    cultivation.json  # 境界基础三围表、五行修为升级表、属性点兑换
    economy.json      # 资源流向、副本次数、每日产出、突破材料需求
    progression.json  # 功能解锁表（什么境界解锁什么）
    visual.json       # 五行色板、特效键、UI 主题 token（见 VISUAL_DIRECTION.md）
  game/               # 领域层：读 config，跑公式
  render/             # 表现层
```

> JSON 为首选格式（策划/非程序员可改、便于热更与后续接服务端下发）。若改用 TS 常量文件，必须 `as const` 且与业务逻辑文件物理分离，不得在同一文件内既定义数值又写规则。

## 4. 加载与校验

1. 启动时由单一 `ConfigService` 统一加载全部配置，**集中一处**，业务层只向它取数。
2. 每张表配一个 schema（Zod / 手写 validator 均可），加载即校验：**字段缺失、类型错误、引用不存在（如关卡引用了不存在的怪/地图）→ 启动即报错**，不允许静默兜底成默认数值。
3. 交叉引用完整性校验：`levels` 引用的 `monsters`/`waves`/`maps`/`rune unlock`、`fusion` 引用的五行精，全部必须能解析。
4. 配置只读：领域层拿到的是冻结（`Object.freeze` / readonly）数据，运行时不得篡改原始配置。

## 5. 公式必须可复现总设计算例

领域层公式实现后，必须能精确复现总设计第五部分算例，作为单元测试基准：

- 焚天符链路：基础 300 → 上品 ×1.7 = 510 → 木区相生 ×1.3 = 663 → 打金怪克制 ×1.5 = 995 → 圆满 2.0 = 1326；被克 ×0.67 = 444。
- 承伤：DEF 150 → 减伤 150/(150+600)=20%，怪攻 100 → 阵眼掉 80。

> 这些数字必须来自“配置喂参 + 公式计算”，不得在测试里硬写期望分支。

## 6. 反面清单（出现任一即判 FAIL）

- 业务 `.ts` 里出现关卡推荐战力、怪物 HP、符攻击、升级消耗等具体平衡数字。
- `switch(关卡ID)` / `if (level === 3)` 这类把关卡内容写进逻辑的分支。
- 新增一个怪/符/关卡需要改业务代码而不是只改配置。
- 配置加载缺少 schema 校验，错误数据被静默兜底。
- 相生/克制/品质倍率等系数散落在多个文件里各写一份。

## 7. 验收门禁挂钩

数值隔离已是 `docs/ACCEPTANCE_GATE.md` L1 的强制阶段，并由 **Claude(监督者)** 机器化实现：

- `npm run check:isolation`（`scripts/check-isolation.mjs`）：扫描领域子目录，硬编码平衡数值 / 引入 Babylon 即 FAIL。已验证有牙。
- 配置 schema 校验：由 Codex 在 Task 1 实现 `ConfigService` + 各表 validator，并写进 **vitest 用例**（非法配置应校验失败、合法应通过、交叉引用可解析），由门禁 `test` 阶段统一执行。
- 门禁脚本与黄金测试归 Claude 维护，Codex 不得擅自放宽（反套娃护栏）。

## 3 · 交付物(精确接口)

### 3.1 `src/game/events/EventBus.ts` — 类型安全总线
```ts
export interface BattleEventMap {
  "battle.prepStarted": { levelId: string };
  "wave.started": { index: number };   // index 从 1 到 wavesPerLevel
  "wave.ended":   { index: number };
  "battle.settled": { victory: boolean; wavesCleared: number; totalSteps: number };
}

export class EventBus {
  on<K extends keyof BattleEventMap>(type: K, handler: (payload: BattleEventMap[K]) => void): () => void; // 返回退订函数
  emit<K extends keyof BattleEventMap>(type: K, payload: BattleEventMap[K]): void;
}
```
- **类型安全**：`emit` / `on` 用错事件名或 payload 形状 → tsc 编译期报错(用 `keyof` + 映射类型实现，不要用 `any`)。
- `on` 返回**退订函数**，调用后不再收到该事件；支持同一事件多个订阅者；对无订阅者的 `emit` 安全(不抛错)。

### 3.2 `src/game/battle/BattleController.ts` — 三段状态机
```ts
import type { GameConfig } from "../../config";
import type { EventBus } from "../events/EventBus";

export type BattlePhase = "prep" | "combat" | "settle";

export interface BattleControllerDeps {
  config: GameConfig;   // 【必须注入】——严禁在类内部调用 loadGameConfig()，否则测试无法喂改过的 config
  bus: EventBus;
  levelId?: string;     // 缺省给个占位关卡 id
}

export class BattleController {
  constructor(deps: BattleControllerDeps);
  get phase(): BattlePhase;
  start(): void;                 // 进入 prep，emit "battle.prepStarted"
  tick(realDtMs: number): void;  // 喂真实帧间隔，按定步长推进
}
```
**相位推进规则(全部数值从 `config.balance.battle` 读)**：
- `start()`：`phase="prep"`，emit `battle.prepStarted`。
- `prep`：累计模拟时间 ≥ `preparationSeconds` → 切 `combat`。
- `combat`：依次推进 `wavesPerLevel` 个**空波次**，每波持续 `defaultWaveIntervalSeconds`；进入每波 emit `wave.started{index}`(index 1..N)，该波时间走完 emit `wave.ended{index}`；最后一波结束 → 切 `settle`。
- `settle`：emit **恰好一次** `battle.settled{ victory:true, wavesCleared:wavesPerLevel, totalSteps }`。**空 battle 无怪攻击阵眼 → 阵眼满血 → 默认胜利**。
- 相位**单向不可逆**；进入 `settle` 后再调用 `tick()` 不得产生任何副作用/重复事件。

### 3.3 `src/game/battle/FixedStepClock.ts` — 定步长累加器(建议)
```ts
export class FixedStepClock {
  constructor(stepMs: number);
  advance(realDtMs: number): number; // 累加真实 dt，返回本次应执行的整数步数；余数留到下次
}
```
- `stepMs = 1000 / config.balance.battle.simulationFps`(由 BattleController 从 config 算好后传入；本类保持纯数学、不读 config)。
- **确定性根基**：把不定的渲染帧间隔切成整数个定长 sim step；相位切换/结算判定都基于**累计 sim step 数**而非真实时间。→ 同一局用 60fps 或 20fps 的帧间隔喂入，`totalSteps` 必须完全一致(架构 §5.1：低端机掉帧不改变战斗结果)。

### 3.4 `src/main.ts` — 最小装配
- `const config = loadGameConfig(); const bus = new EventBus(); const battle = new BattleController({ config, bus });`
- 接进 BabylonApp 的渲染/定步长循环：每帧 `battle.tick(engine.getDeltaTime())` 之类(**装配层可 import Babylon**)。
- **最小改动**，不破坏现有引导；不把任何玩法规则/数值写进 main.ts。

## 4 · 关键设计(为何这样定)
1. **构造注入 config，而非内部 `loadGameConfig()`**：锚点测试要喂一个把 `simulationFps` 翻倍的 config，验证步数随之变化以证明"tick 率读 config、非代码写死 30"。内部直读会让这条不可测 → 判 FAIL。
2. **定步长 = 可复现 = 可平衡校准**：结算与相位切换只认 sim step 计数，真实 dt 只用于喂累加器。这样回放/埋点/平衡模拟器都能复现同一局。
3. **"空"的边界**：本批 COMBAT 只做空波次时间推进(占位)，**不刷真怪、不算伤害、不扣阵眼血**。事件命名(`wave.*`、`battle.settled`)对齐架构 §4 事件表，为 T4(地图)/T6(怪·符·承伤)预留接线，届时把空波次替换为真实 WaveScheduler。
4. **事件即埋点源**：EventBus 广播的事件同时是未来 TelemetryService 的数据源，一套事件两用。

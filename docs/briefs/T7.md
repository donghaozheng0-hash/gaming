## 3 · 交付物要点(接口细节以尺子头注为准,此处只补 UI 与装配)

### 3.1 战斗逻辑层(全部无 DOM,headless 可测)
按尺子头注 1)–6) 逐条实现。注意:
- `submitDraw` 冷却是**模拟步数**口径(秒×simulationFps 取整),不是真时钟;boost 只吃该符下一发。
- 融合后该符的 kezhiMul 改查融合克制表(advantage.target→multiplier / disadvantage.source→multiplier / 其余 1.0),**不再走五行 relation()**;xiangsheng 档位冻结为融合前取值。
- `RunProgression` 的 rng 消耗顺序(非保底波:chance→命中才 roll 元素;保底波:只 roll 元素)是确定性回放的契约,别调换。
- 债#5:CombatSimulation 需按出生波跟踪"该波全部 spawned 怪是否已 died∪leaked",settle 时输出完全清场口径 wavesCleared 与计时器口径 wavesDispatched。

### 3.2 UI 层(首版,雾山五行基调按 T7.0 拍板稿)
- **PREP 情报条(R4)**:备战期顶部横条,渲染 `summarizeWaveElements`:五行图标(带描边,金要衬底)+潜在数量点+Boss 波朱砂旗;`wave.started` 后收起为小标签。
- **灵机 HUD+菜单**:右下角灵机墨点(亮=可用);点开菜单二选一:①升级(列出战符,点选→`run.upgradeRune`)②融合(列 unlockSchedule[0] 三配方,材料/元素不满足置灰并注明缺什么——**UI 预检,不靠 catch throw**)。
- **画符层**:点"画符"→选符→全屏半透明符纸,按住拖拽记录轨迹(pointer 事件,坐标归一化)→松手 `scoreStroke(points, rune.drawTemplate)`→`sim.submitDraw`→按 T7.0 draw 板块三档样式反馈(涣散/成符/朱边/金边)；冷却中入口置灰+剩余秒数。
- **融合标记**:融合成功的符位标记合成元素(雷=紫金示意,颜色进 visual.json palette 扩展)。
- 视觉参数(尺寸/间距/时长/颜色)全进 `visual.json`(+schema/visual.ts 同步有牙),颜色纪律静态扫描零硬编码。
- 全部交互挂在游戏画布层,与调参台(?debug 默认开)并存不互挡;`?debug=0` 纯游戏画面一切可用。

### 3.3 main.ts 装配
`assembleBattle` 产物上挂 `RunProgression`,UI 层注入 {config, bus, simulation, run, loadout};草图板/调参台守门不动。

## 4 · 关键设计(为何这样定)
1. **交互先行、特效后置**:本批把"资源→决策→数值变化"闭环钉死(可测),T7b 才做特效引擎——把不可测的表现和可测的数值分两批验收。
2. **三个入口都走 runeDamage 单公式**(升级=base/融合=kezhi/画符=drawBonus),公式层保持唯一伤害真相。
3. **rng 消耗顺序入契约**是为了确定性回放(调参台 replay 将来要回放带成长的局)。
4. **UI 预检置灰**而非弹错:摸鱼玩家的手感底线(2.7 零惩罚精神)。

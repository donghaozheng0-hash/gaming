## 3 · 交付物(精确接口)

### 3.1 `src/game/battle/combat/CombatSimulation.ts`(纯领域,无 Babylon)

接口签名以 `combat.gate` 头注为准(那是唯一权威)。要点:

```ts
export interface CombatLoadoutEntry { slotIndex: number; runeId: string }
export interface CombatSnapshot { coreHp; coreMaxHp; kills; leaks; monstersAlive; lootMultiplier }
export interface CombatSimulationDeps {
  config: GameConfig; map: GeneratedMap; bus: EventBus;
  rng: () => number; loadout: readonly CombatLoadoutEntry[];
  requiredPower: number; waveTemplateId?: string; // 缺省 config.waves.waveTemplates[0].id
}
export class CombatSimulation {
  constructor(deps: CombatSimulationDeps);
  spawnWave(waveIndex: number): void; // 模板外 index → throw
  step(): void;                        // 1 定步 = 1/config.balance.battle.simulationFps 秒
  get coreHp(): number;
  isFieldCleared(): boolean;
  snapshot(): CombatSnapshot;
}
```

数值来源对照(全部注入,零裸数字):
- 怪 HP/盾/攻 = `monsters.json 系数 × requiredPower`;速度 = `speedUnitsPerSecond`(canvas units/s)。
- 阵眼:`basePower = requiredPower × balance.playerDerivation.basePowerFrac`;
  `u = basePower / (Σ statRatio[k] × powerFormula[k+'Weight'])`(= atk×6 + hp×0.6 + def×3 权重和);
  `coreMaxHp = u × statRatio.hp`,`coreDEF = u × statRatio.def`。
- 漏怪承伤 = `coreDamage({ atk: 怪攻, def: coreDEF, relK: balance.defense.relK })`(T2 公式,直接调用)。
- 符伤 = `runeDamage({ base: rune.lv1Attack, qualityMul: balance.damageFormula.qualityMultipliers.xia_pin, xiangshengMul: 两档判定, kezhiMul: balance.damageFormula.elementalMultipliers[relation(rune.element, monster.element)], drawBonus: balance.damageFormula.drawBonus.base })`。
  T6 全下品(×1)、无画功(base=0)——**用配置键取值,不写 1 和 0 的字面量**。
- R1 两档判定(每符静态,摆位时算一次即可):
  存在其他 loadout 格 j,`openSlots[j]` 与本格距离 ≤ `balance.battle.xiangshengAdjacencyMaxCanvasUnits`,且 j 上符五行**生**本符五行(`balance.elements.xiangshengCycle[elemJ] === elemI`)→ `xiangshengMultipliers.generated`;
  否则任意场上符生本符 → `.presence`;否则 `.neutral`。
- 有效射程 = `min(rune.range.value, slotType.rangeUnits)`;`range.kind==="global"` → `slotType.rangeUnits`(slotType 由 `openSlots[i].slotTypeId` 查 `maps.candidateSlotTypes`)。
- 冷却 = `Math.round(simulationFps / rune.attackSpeedPerSecond)` 步,首发即可击(冷却初始为 0)。
- 索敌:候选 = 存活且与本格欧氏距离 ≤ 有效射程的怪;策略 = `resolveTargetingStrategy(rune.targetingStrategyId)`。
- 怪推进:沿 `map.routes[routeIndex]` **折线几何长度**推进,每步 `speed/simulationFps`;多路线模板每只怪 spawn 时用 `rng` 均匀选 routeIndex;走完 → leaked(不算 kill)。
- 波入场:`spawnWave(i)` 登记模板第 i 波;之后每 step 按 `spawnIntervalSeconds`(null 视为 0=同刻)逐只入场;entry 的 `monsterPoolIds` 多 id 每只 `rng` 均匀抽;怪五行从 `defaultElements` 均匀抽。**rng 消费顺序注意确定性**(建议:每只怪按 [池选择,路线,五行] 固定顺序抽)。
- `onDeath`:死亡时按 `spawnMonsterId/count/hpCoefficientR` 在**死亡位置**入场子怪(distance 继承母怪),emit `monster.spawned`。
- `lootMultiplier = infinite.lootCompensation.byOpenSlotCount[String(map.openSlots.length)] ?? 1`。
- entityId:单局内从 1 递增。盾先吸伤,溢出转血。

## 4 · 关键设计(为何这样定)

1. **CombatSimulation 独立于 BattleController 且可选注入**:T3 空转契约是已验收资产,注入式扩展保证向后兼容;sim 純领域无 Babylon,尺子能 headless 全量验证(掉帧不改结果的定步长架构延续)。
2. **策略词条化(R6)**:目标选择做成注册表而非 if/else,后续"最远/残血/优先精英"词条只加数据+注册函数,不动战斗核心——加构筑深度不加操作负担(底层逻辑一)。
3. **R1 两档相生**:2-3 开放格下相邻相生几率有限(相邻阈值 200 时约 13-25%),同场弱档(×1.1)保证相生系统在任何开格拓扑下有效,相邻强档(×1.3)保留摆位技巧上限;系数经 balance-sim 六曲线校准,预算=技巧补偿 30% 上限,别改。
4. **漏怪扣血是设计内张力**:容量模型显示 2 格局末波压力比≈1.0,漏 1-2 只、阵眼扛住是**预期体验**(阵眼血=容错资源);所以 leaked 不判失败,只有 coreHp 归零才败;R3 的 ×1.25 补偿即为此风险定价。
5. **几何长度 ≠ 名义 1000**:三模板折线实长 820-1480,怪按真实几何走,渲染与逻辑一个世界;数值模型的 1000 只是解析近似,已在 balance-sim 剧本里,不要把 1000 写进战斗代码。
